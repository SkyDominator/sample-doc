# 핵심 개념

`nano-llm-engine`는 실제 하드웨어 스택을 단순화한 모의 SDK이지만, 문서 구조와 검증 전략은 실제 추론 런타임 문서화 방식과 동일하게 설계되었습니다.
이 문서는 API 호출 자체보다, 왜 이런 옵션이 존재하는지와 운영 관점에서 어떤 의미가 있는지에 초점을 둡니다.

## 양자화(Quantization)

양자화는 모델 가중치와 연산 정밀도를 낮춰 메모리 사용량과 지연 시간을 줄이는 전략입니다.

- `FP32`: 가장 높은 정밀도, 가장 큰 메모리 비용
- `FP16`: 일반적으로 정확도-성능 균형이 좋음
- `INT8`: 추론 가속과 메모리 절약 효과가 큼
- `INT4`: 메모리 효율은 매우 좋지만 품질 저하 가능성이 큼

문서 관점에서는 "어떤 정밀도가 빠르다"보다 "품질과 처리량의 트레이드오프를 어떤 기준으로 선택하는가"를 명확히 안내해야 합니다.

## KV Cache 메모리 모델

LLM 추론에서 KV cache는 이전 토큰의 attention key/value 상태를 저장해 재계산을 줄입니다.
`KVCacheConfig.max_seq_len`이 커질수록 긴 문맥을 처리하기 유리하지만 메모리 점유가 커집니다.

실무 문서에서는 다음과 같은 의사결정 지점을 분리해 설명하는 것이 좋습니다.

- 긴 컨텍스트 정확도 우선: `max_seq_len` 증가
- 동시 요청 처리량 우선: `max_seq_len` 축소 + 배치 튜닝
- 메모리 한계 환경: 낮은 정밀도 + 보수적 캐시 크기

## 배치 추론(Batched Inference)

`InferenceConfig.batch_size`는 동일 시간 구간에 묶어 처리하는 요청 수를 의미합니다.
배치가 커질수록 디바이스 활용률은 증가하지만 개별 요청 지연(latency)은 증가할 수 있습니다.

문서 작성 시 권장되는 표현:

1. 실시간 API: 낮은 배치, 낮은 지연
2. 오프라인 처리: 높은 배치, 높은 처리량
3. 혼합 워크로드: 시간대별 배치 정책 분리

## 토큰 생성과 스트리밍

토큰 생성은 매 스텝마다 다음 토큰 분포를 계산하는 반복 과정입니다.
스트리밍 출력은 최종 문자열 완성 전에도 중간 결과를 사용자에게 전달해 체감 응답성을 높입니다.

```python
import os

from nano_llm import InferenceConfig, NanoLLMEngine

os.environ["MOCK_RNGD_HARDWARE"] = "true"
engine = NanoLLMEngine(model_path="models/mock-llm.bin", quantization="INT8")
engine.load_to_device()

for token in engine.generate_streaming(
    "토큰 스트리밍의 장점을 세 가지로 요약해줘",
    InferenceConfig(max_tokens=40, temperature=0.1),
):
    print(token)
```

## 문서 검증과 드리프트 방지

이 프로젝트는 코드 샘플을 CI에서 직접 실행해 문서와 코드 간 괴리를 조기에 감지합니다.
이 방식은 특히 빠른 릴리즈 주기에서 중요합니다.

- API 시그니처 변경 -> `generate_api.py` 재생성 결과로 즉시 노출
- 샘플 코드 파손 -> `validate_snippets.py`에서 빌드 실패
- 용어/표현 불일치 -> 번역 파이프라인에서 glossary 보호

핵심은 문서를 산출물이 아니라 테스트 가능한 제품으로 다루는 것입니다.
