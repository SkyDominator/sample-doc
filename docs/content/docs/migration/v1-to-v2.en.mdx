# v1 to v2 Migration

This guide summarizes what to check when moving from a v1 style centered on minimal constructor calls to a v2 style that separates engine configuration from request configuration.

## Change summary

- Keep the `quantization` default, but set it explicitly in the constructor when needed
- Move cache-related settings to engine level through `KVCacheConfig`
- Pass per-request token budget and sampling options through `InferenceConfig`
- Reuse the same request configuration across synchronous and streaming calls

## Before (v1 style)

```python
import os
from nano_llm import NanoLLMEngine

os.environ["MOCK_RNGD_HARDWARE"] = "true"
engine = NanoLLMEngine(model_path="models/mock-llm.bin")
engine.load_to_device(0)
print(engine.generate("hello"))
```

## After (recommended v2 style)

```python
import os
from nano_llm import InferenceConfig, KVCacheConfig, NanoLLMEngine

os.environ["MOCK_RNGD_HARDWARE"] = "true"
engine = NanoLLMEngine(
    model_path="models/mock-llm.bin",
    quantization="INT8",
    kv_cache_config=KVCacheConfig(max_seq_len=4096, cache_dtype="fp16"),
)
engine.load_to_device(0)
print(engine.generate("hello", InferenceConfig(max_tokens=32, temperature=0.0)))
```

## Step-by-step migration

1. Add imports for `InferenceConfig` and `KVCacheConfig`.
2. Pass `kv_cache_config=KVCacheConfig(...)` when creating the engine.
3. Pass `InferenceConfig` to both `generate()` and `generate_streaming()`.
4. Check that the returned string reflects the selected `quantization` and `max_tokens` values.

## Regression checklist

- Does loading without `MOCK_RNGD_HARDWARE` still raise `DeviceNotFoundError`?
- Does `engine.kv_cache_config` store the expected cache configuration?
- If you join `generate_streaming()` tokens back together, do you get the same output as the synchronous call?
- After `unload()`, are the loaded state and memory telemetry reset?
