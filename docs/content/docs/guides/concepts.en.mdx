# Core Concepts

`nano-llm-engine` is an inference runtime example that mocks RNGD device integration.
It does not perform real model computation, but it recreates a small control surface around a runtime, including engine lifecycle, request configuration, and memory abstractions.

## Quantization

`quantization` is a literal value stored on the engine instance, and it is also reflected directly in the current mock output prefix.

- `FP32`: highest-precision label
- `FP16`: default value
- `INT8`: label for a memory-saving configuration
- `INT4`: label for more aggressive compression

In this codebase, quantization does not change numeric computation, but it lets you track which configuration produced a request by inspecting the output string alone.

## KV Cache Memory Model

In LLM inference, the KV cache is a buffer that preserves attention key/value state from previous tokens to avoid recomputation.
`KVCacheConfig` stores the assumptions for this buffer on the engine.

- `max_seq_len`: maximum sequence length the cache is expected to hold
- `cache_dtype`: precision label for cache storage

The current mock engine does not apply `KVCacheConfig` directly to memory calculations, but it explicitly preserves which cache policy the engine instance was created with.

## Batched Inference

`InferenceConfig` provides request-scoped controls.

- `batch_size`: number of prompts grouped into one request
- `max_tokens`: token budget reflected in the output prefix
- `temperature`: placeholder value representing sampling diversity

In this implementation, `generate()` exposes only `max_tokens` in the string output, but `batch_size` and `temperature` are still passed together to preserve a runtime-like call shape.

## Device Memory Abstraction

`NanoLLMEngine.get_device_memory()` returns simple telemetry based on whether the engine is loaded.
Before loading, `used_mb=0`; after loading, `used_mb=1024`.

A separate `DeviceMemory` wrapper assumes an independent memory pool.

- `allocate(bytes_size)`: returns `True` if reservation succeeds, or `False` if capacity is insufficient
- `free(bytes_size)`: raises `RuntimeError` if more bytes are released than were reserved
- `utilization()`: returns the current utilization in the range `0.0` to `1.0`

If the Rust extension module `nano_llm_rs` is available, its implementation is used; otherwise the wrapper falls back to the Python implementation automatically.

## Token Generation and Streaming

`generate()` constructs a string with the prefix `[mock:{quantization}:tokens={max_tokens}]`.
`generate_streaming()` splits the same result on whitespace and yields tokens in order.

```python
import os

from nano_llm import InferenceConfig, NanoLLMEngine

os.environ["MOCK_RNGD_HARDWARE"] = "true"
engine = NanoLLMEngine(model_path="models/mock-llm.bin", quantization="INT8")
engine.load_to_device()

for token in engine.generate_streaming(
    "Explain how the token budget is reflected in the output format under the INT8 setting",
    InferenceConfig(max_tokens=40, temperature=0.1),
):
    print(token)
```

## Tested Behavior Surface

`sdk/tests/test_engine.py` validates the core contract of the current mock runtime.

- Missing environment variables cause `load_to_device()` to raise `DeviceNotFoundError`
- Calling `generate()` before load raises `GenerationError`
- Streaming output delivers the synchronous generation result token by token
- Memory telemetry changes with engine load state
- `DeviceMemory` performs allocation, release, and utilization calculations consistently
