# 추론 파이프라인

이 튜토리얼은 실제 서비스 문서처럼 "설정 -> 로드 -> 생성 -> 스트리밍 -> 메모리 점검 -> 오류 처리 -> 종료" 순서로 구성되어 있습니다.
모든 코드는 CI에서 실행 가능한 형태를 유지합니다.

## 1) 런타임 초기화

```python
import os

from nano_llm import InferenceConfig, KVCacheConfig, NanoLLMEngine

os.environ["MOCK_RNGD_HARDWARE"] = "true"

engine = NanoLLMEngine(
    model_path="models/mock-llm.bin",
    quantization="INT8",
    kv_cache_config=KVCacheConfig(max_seq_len=8192, cache_dtype="fp16"),
)
engine.load_to_device(device_id=0)
```

## 2) 단건 생성 요청

```python
from nano_llm import InferenceConfig

cfg = InferenceConfig(batch_size=1, max_tokens=80, temperature=0.2)
out = engine.generate("SDK 문서 자동 검증의 장점 세 가지를 알려줘", cfg)
print(out)
```

`generate()` 반환 문자열 앞에는 양자화 라벨과 토큰 예산 메타데이터가 함께 붙어, 문서 예제만 봐도 어떤 설정으로 생성되었는지 바로 확인할 수 있습니다.

## 3) 스트리밍 요청

```python
from nano_llm import InferenceConfig

for token in engine.generate_streaming(
    "배치 추론과 지연 시간의 관계를 요약해줘",
    InferenceConfig(max_tokens=40, temperature=0.1),
):
    print(token)
```

`generate_streaming()`은 동기 생성 결과를 공백 기준 토큰 단위로 나눠 전달하므로, 문서에서는 최종 문자열과 스트리밍 조각이 같은 설정에서 나온다는 점을 함께 설명하는 것이 좋습니다.

## 4) 메모리 텔레메트리 확인

```python
stats = engine.get_device_memory()
print(stats["total_mb"], stats["used_mb"])
```

`get_device_memory()`는 엔진이 현재 점유 중인 mock 메모리 상태를 읽습니다. 배치 조정이나 KV cache 길이 조정 전에 현재 사용량을 먼저 확인하는 흐름을 권장합니다.

## 5) 장치 메모리 풀 예약 확인

```python
from nano_llm import DeviceMemory

memory = DeviceMemory(total_bytes=32 * 1024 * 1024)
reserved = memory.allocate(16 * 1024 * 1024)
print(reserved, f"{memory.utilization():.2f}")

memory.free(16 * 1024 * 1024)
print(f"{memory.utilization():.2f}")
```

이 단계는 엔진 텔레메트리 조회와 별개로, 새 요청을 투입하기 전에 메모리 예산을 사전 점검하는 패턴을 보여줍니다.

## 6) 오류 처리 패턴

```python
import os

from nano_llm import NanoLLMEngine
from nano_llm.errors import DeviceNotFoundError

os.environ.pop("MOCK_RNGD_HARDWARE", None)

probe = NanoLLMEngine(model_path="models/mock-llm.bin")
try:
    probe.load_to_device(0)
except DeviceNotFoundError as exc:
    print(f"recoverable error: {exc}")
```

## 7) 종료

```python
engine.unload()
```
